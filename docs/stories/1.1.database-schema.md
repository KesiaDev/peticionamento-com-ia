### Naming Conventions para Migrations
- Formato Lovable-compatible: `00001_description.sql` (numeração sequencial simples)
- NÃO usar timestamps como nome (incompatível com Lovable Cloud)

### Pattern: Updated At Trigger
```sql
CREATE OR REPLACE FUNCTION public.set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```
Reutilizar esta function em todas as tabelas que tenham `updated_at`.

### Pattern: RLS com organization_id
Todas as policies de SELECT devem usar subquery em `profiles` para determinar a organização do usuário autenticado:
```sql
auth.uid() IN (
  SELECT p.id FROM profiles p
  WHERE p.organization_id = {table}.organization_id
)
```

### Issues Found

| # | Severity | Description | Recommendation |
|---|----------|-------------|----------------|
| 1 | MEDIUM | `handle_new_user()` slug generation can produce invalid slugs. If `full_name` contains only special characters, after `regexp_replace` and `trim`, the base slug is empty, resulting in a slug like `-a1b2c3d4` which violates the `slug_format` CHECK constraint (`^[a-z0-9]...`). | Add a fallback: `IF _slug = '' THEN _slug := 'org'; END IF;` before appending the UUID suffix. |
| 2 | LOW | Duplicate index on `slug`. The `UNIQUE` constraint already creates a unique index. The explicit `CREATE INDEX idx_organizations_slug` adds a redundant non-unique index. | Remove the explicit `CREATE INDEX` on slug, or keep it as documentation -- no functional harm. |
| 3 | LOW | No DELETE policies on `organizations` or `profiles`. Authenticated users cannot delete orgs or remove members via the API. | Acceptable if deletion is service_role-only by design. Add a comment in the RLS migration documenting this decision. |
| 4 | LOW | `handle_new_user()` does not validate that a provided `organization_id` exists before inserting into profiles. The FK will raise a generic constraint error. | Consider adding an existence check with a user-friendly `RAISE EXCEPTION` message, or document that the FK error is the expected behavior. |
| 5 | LOW | Minor spec inconsistency: AC2 says `branding JSONB DEFAULT '{}'::jsonb` but the migration uses the full JSON object from Dev Notes. | Not a real issue -- Dev Notes take precedence and the full default is better. |
