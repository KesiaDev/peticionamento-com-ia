## Story
**As a** usuário,
**I want** fazer login e registro na plataforma,
**so that** eu possa acessar as funcionalidades do sistema.

6. Componente `ProtectedRoute` que redireciona para `/login` se não autenticado

8. Após login, redireciona para `/dashboard`

### Issues Found

#### MEDIUM-1: Race condition between getSession and onAuthStateChange
**File:** `src/contexts/AuthContext.tsx` (lines 52-85)
**Description:** On mount, `getSession()` and `onAuthStateChange` can both fire for the same SIGNED_IN event, causing duplicate `fetchProfileAndOrganization` calls. The Supabase docs recommend setting up the listener first, then calling `getSession`, or guarding against double-fetch.
**Recommendation:** Either (a) set up `onAuthStateChange` before `getSession` and use a ref to deduplicate, or (b) rely solely on `onAuthStateChange` which fires on mount with INITIAL_SESSION event in newer Supabase versions, removing the explicit `getSession` call.

#### MEDIUM-2: Register may navigate before profile/org exist
**File:** `src/pages/Register.tsx` (line 70), `src/contexts/AuthContext.tsx`
**Description:** After `register()` succeeds, the component immediately calls `navigate("/dashboard")`. However, profile and organization are created server-side (via Supabase trigger/function), and `onAuthStateChange` fires asynchronously. The Dashboard page may render with `profile = null` and `organization = null` briefly, showing fallback values ("Usuario", "JurisTech"). This is a UX concern, not a crash.
**Recommendation:** Consider adding a brief loading state on Dashboard until profile is loaded, or await the auth state change before navigating.

#### MEDIUM-3: Missing accented characters in validation messages
**Files:** `src/pages/Login.tsx` (line 28), `src/pages/Register.tsx` (lines 31, 38)
**Description:** Portuguese validation messages are missing diacritical marks: "invalido" should be "invalido" (though the story Dev Notes show the same pattern without accents). UI text like "Nao tem conta?" (line 125, Login.tsx) and "Ja tem conta?" (line 189, Register.tsx) also lack accents. While this matches a deliberate ASCII-only pattern in the codebase, it degrades the professional appearance for a Brazilian Portuguese product.
**Recommendation:** If ASCII-only is intentional (e.g., to avoid encoding issues), document this decision. Otherwise, add proper accents: "invalido" -> "invalido", "Nao" -> "Nao", etc.

#### LOW-1: Unit tests not implemented (Task 7)
**Files:** `src/test/auth/` directory does not exist
**Description:** All 5 test subtasks (7.1-7.5) are unchecked. The story acknowledges this in Completion Notes. Per the story definition, tests are part of scope.
**Recommendation:** Tests should be created in a follow-up or the story should explicitly defer them. Not blocking since the Dev Notes acknowledge this.

#### LOW-2: Task 1.4 (.env verification) unchecked
**Description:** The task to verify `.env` has the correct Supabase variables is unchecked. The `.env.example` file has the variables defined, and the `supabase.ts` client throws at runtime if they are missing, which is good defensive coding. However, there is no `.env` file to verify (only `.env.example`).
**Recommendation:** Mark as N/A or verify in deployment documentation.

#### LOW-3: AuthProvider placement relative to BrowserRouter
**File:** `src/App.tsx` (lines 20-24)
**Description:** The story spec (Task 2.3) says AuthProvider should be "dentro do QueryClientProvider, fora do BrowserRouter". The implementation places AuthProvider outside BrowserRouter, which matches the spec. However, AuthContext functions (`login`, `register`) don't use navigation — the pages handle their own navigation via `useNavigate`. This is correct. No issue, just confirming alignment.
