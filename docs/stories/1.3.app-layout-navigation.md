## Story
**As a** usuário autenticado,
**I want** ver um layout com sidebar e navegação,
**so that** eu possa acessar os módulos do sistema de forma organizada.

## Acceptance Criteria
1. Componente `AppLayout` com sidebar colapsável, top bar com avatar/nome e área de conteúdo
2. Sidebar com links para: Dashboard, IA Jurídica, Clientes, Processos, Tarefas, Financeiro, Configurações
3. Ícones Lucide para cada item do menu
4. Top bar com nome do usuário, role badge e botão de logout
5. Layout responsivo: sidebar collapsa em mobile (hamburger menu)
6. Rotas protegidas: `/dashboard`, `/ai/*`, `/clients/*`, `/cases/*`, `/tasks`, `/finance`, `/settings`
7. Página Dashboard com placeholder "Em construção" e mensagem de boas-vindas
8. Breadcrumb navigation funcional

### Issues Found

#### Severity: MEDIUM

1. **Missing diacritics/accented characters in all user-facing strings**
   - `navigation.ts`: `"IA Juridica"` should be `"IA Juridica"` -> `"IA Jurídica"`, `"Configuracoes"` -> `"Configurações"`, `"Usuarios"` -> `"Usuários"`, `"Proprietario"` -> `"Proprietário"`, `"Secretaria"` -> `"Secretária"`, `"Estagiario"` -> `"Estagiário"`
   - `Dashboard.tsx`: `"Organizacao"` -> `"Organização"`, `"construcao"` -> `"construção"`, `"Graficos, relatorios e metricas estarao disponiveis"` -> `"Gráficos, relatórios e métricas estarão disponíveis"`
   - `Placeholder.tsx`: `"construcao"` -> `"construção"`, `"estara disponivel"` -> `"estará disponível"`
   - `App.tsx`: `"IA Juridica"` -> `"IA Jurídica"`, `"Configuracoes"` -> `"Configurações"`, `"Usuarios"` -> `"Usuários"`
   - `AppLayout.tsx`: `"navegacao"` -> `"navegação"` (SheetTitle sr-only)
   - **Impact:** All PT-BR text displayed to users is missing accents throughout the implementation. This is a UX issue for a Brazilian Portuguese application.
   - **Fix:** Add proper UTF-8 accented characters to all strings listed above.

2. **`roleLabels` includes `"secretary"` but `UserRole` type only has `"owner" | "admin" | "lawyer" | "intern"`**
   - File: `src/config/navigation.ts` line 44 vs `src/types/database.types.ts` line 1
   - **Impact:** The `secretary` label will never match a real role from the database. Dead code / spec mismatch.
   - **Fix:** Either add `"secretary"` to `UserRole` type (if the role exists in the DB enum) or remove it from `roleLabels` to avoid confusion. Verify against the Supabase `app_role` enum.

#### Severity: LOW

3. **`/settings/*` wildcard route missing from App.tsx**
   - AC6 specifies `/settings/*` but only `/settings`, `/settings/users`, and `/settings/profile` are defined as explicit routes.
   - **Impact:** Any future settings sub-route (e.g., `/settings/billing`) will hit the 404 NotFound page instead of a placeholder. Current explicit routes cover the documented cases, so this is low severity.
   - **Fix:** Consider adding `<Route path="/settings/*" element={<Placeholder title="Configurações" />} />` as a catch-all after the specific settings routes, or leave as-is and add routes incrementally.

4. **Tests not implemented (Task 8)**
   - All 5 test subtasks (8.1-8.5) are unchecked.
   - **Impact:** No automated test coverage for layout components. The story notes indicate tests are pending and require QA gate, so this is expected at this stage.
   - **Fix:** Tests should be created before story is marked Done. This is tracked and not blocking the CONCERNS verdict.

5. **`NavLink` key duplication risk in non-collapsed mode**
   - In `Sidebar.tsx`, the `key` prop is set on the `NavLink` inside the `link` variable (line 46), but when not collapsed, the `link` JSX is returned directly without a wrapping element with a key. React uses the `key` from `NavLink` which works, but the pattern is slightly inconsistent between collapsed (key on `Tooltip`) and non-collapsed (key on `NavLink` inside `link`).
   - **Impact:** Functionally works. No runtime error. Minor code quality observation.

### Summary

The implementation is solid and covers all 8 acceptance criteria. The component architecture is clean: navigation config is centralized, layout uses proper React Router nesting with Outlet, mobile responsiveness uses the existing useIsMobile hook with Sheet for drawer behavior, and breadcrumbs are dynamically generated from the URL.

The primary concern is the **systematic absence of diacritical marks** in all Portuguese strings, which affects the user-facing quality of a Brazilian Portuguese application. This should be fixed before the story is marked Done.

The `secretary` role label mismatch with the type system should also be clarified.

### Recommendation

Fix the diacritics issue (medium severity) and clarify the secretary role before moving to Done. Tests (Task 8) should be implemented as a follow-up.
