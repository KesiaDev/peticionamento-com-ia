## Story
**As a** advogado (lawyer),
**I want** que a IA busque jurisprudencia relevante nos tribunais brasileiros,
**so that** meus documentos tenham embasamento em decisoes reais e atualizadas.

## Acceptance Criteria
1. Edge Function `tribunal-search` que consulta APIs/sites dos tribunais (STF e STJ possuem APIs publicas REST; TJPE via web scraping como fallback)
2. Busca por palavras-chave em: STF, STJ e TJPE (prioridade alta). Comecar por STF/STJ (APIs documentadas), TJPE como segunda fase
3. Resultados retornam: numero do processo, ementa, data, tribunal, link
4. IA incorpora jurisprudencia relevante no documento gerado automaticamente
5. Usuario pode adicionar/remover jurisprudencias antes de finalizar
6. Cache de resultados por 24h para economia de requisicoes
7. Fallback graceful se tribunal estiver indisponivel

## Dev Notes

### APIs dos Tribunais Brasileiros

**STF (Supremo Tribunal Federal):**
- API REST publica documentada
- Base URL: `https://jurisprudencia.stf.jus.br/api/search/acervo`
- Parametros: `q` (query), `page`, `pageSize`
- Retorna JSON com acervo de jurisprudencia
- Rate limit: nao documentado oficialmente, usar throttle de 1 req/s

**STJ (Superior Tribunal de Justica):**
- API REST publica
- Base URL: `https://scon.stj.jus.br/SCON/`
- Endpoint de pesquisa: `https://scon.stj.jus.br/SCON/pesquisar.jsp`
- Suporta busca por palavras-chave, relator, orgao julgador
- Formato de resposta: HTML/JSON dependendo do accept header

**TJPE (Tribunal de Justica de Pernambuco):**
- NAO possui API publica REST
- Necessita web scraping como fallback
- URL de busca: `https://www.tjpe.jus.br/jurisprudencia`
- Scraping sujeito a quebras por mudanca de layout — implementar como segunda fase
- Considerar puppeteer/playwright se HTML statico nao funcionar (avaliar peso no Edge Function)

### Arquitetura
- Edge Function e obrigatoria pois API keys e secrets nao devem estar no frontend (NFR4)
- Cache no banco Supabase com RLS por organization_id (NFR5)
- Timeout maximo de 30s para a Edge Function completa (NFR6: < 30s para geracao IA)
- Considerar paralelizar chamadas aos tribunais com `Promise.allSettled()`

### Implementation Notes
- STF and STJ clients attempt real API calls first, falling back to mock data if the API response format is unexpected or the API is unreachable
- TJPE uses mock data as placeholder since no public REST API exists (web scraping deferred to second phase)
- Mock data contains realistic Brazilian jurisprudence entries with proper formatting
- Input validation done manually (no Zod in Deno Edge Functions to avoid import issues), but validates same constraints
- Cache uses SHA-256 hashing of query+court for cache keys
- Stale-while-revalidate: if a court API fails, expired cache entries are served with a warning
- JurisprudenceSearch component integrated into DocumentWizard Step 2, injecting selected jurisprudence into the `legal_basis` field of the generation context

### Code Quality Issues

1. **BUG (Low):** `JurisprudenceSearch` receives `initialSelected` prop but never uses it to seed the hook's internal `selectedResults` state. The `isSelected` function checks `initialSelected` for display, but handleRemove/handleAdd operate only on the hook's state, causing a desync if `initialSelected` is non-empty.
2. **BUG (Low):** In `handleAdd`, `onSelectionChange` is called with `[...selectedResults, result]`, but `selectedResults` is the stale value before `addResult` updates state. The parent will receive the correct list only on the next render. Same issue in `handleRemove`.
3. **TECH DEBT:** Edge Function falls back silently to mock data on any API error (line 213), making it impossible to distinguish "real results" from "mock results" in production. Should at least include a warning when mock data is returned.
4. **TECH DEBT:** Cache upsert uses delete-then-insert (lines 117-129) instead of a single SQL upsert, introducing a small race condition window.
5. **GOOD:** Validation matches story spec (query min 3 chars, courts enum, limit 1-50). Service role used for cache writes. Auth enforced.
