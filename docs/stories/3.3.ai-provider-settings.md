## Story
**As a** admin,
**I want** configurar qual provedor de IA minha organizacao usa,
**so that** eu possa controlar custos e escolher o modelo mais adequado para o escritorio.

## Acceptance Criteria
1. Secao em `/settings/integrations` para configuracao de IA
2. Seletor de provider padrao: OpenAI, Gemini, Claude
3. Campo para API key do provider (salva encriptada no banco)
4. Seletor de modelo por provider (ex: gpt-4o, gpt-4o-mini, gemini-pro, claude-sonnet)
5. Teste de conexao: botao "Testar" que faz chamada simples e confirma
6. Exibicao de uso mensal (tokens consumidos, custo estimado)
7. Limite de uso configuravel por plano

- API keys NUNCA devem trafegar ou ser armazenadas em texto claro no frontend (NFR4)
- Encriptacao com `pgcrypto` usando symmetric key armazenada como secret do Supabase (variavel de ambiente `ENCRYPTION_KEY`)

### Code Quality Issues

1. **SECURITY (High):** API key is stored and returned in plaintext via `organizations.llm_config`. The `maskApiKey` function only masks in the placeholder UI text, but `config.api_key` is fully readable in the frontend state and network responses. This violates AC3 ("salva encriptada no banco") and NFR4.
2. **BUG (Medium):** `handleSave` sends `api_key: apiKey || (config?.api_key ?? "")` — when user leaves the field blank to keep existing key, it re-sends the full plaintext key from the fetched config. If config fetch ever returns masked key, this would overwrite with the mask.
3. **MISSING:** No Zod validation for API key format (Task 1.6 open).
4. **MISSING:** No dedicated `test-ai-connection` Edge Function (Tasks 3.1, 3.2 unchecked). Reuses `ai-generate` which works but does not isolate test logic.
5. **MISSING:** No `ai_provider_config` table (Task 2.1-2.5 all unchecked). Config piggybacks on `organizations.llm_config`.
6. **MISSING:** No `ai_usage_log` table migration (Task 4.1-4.2 unchecked). `fetchUsageStats` will throw if table does not exist.
7. **MISSING:** No usage limit enforcement (Task 4.8). No daily usage chart (Task 4.6). No RLS for usage log (Task 4.9).
8. **GOOD:** ProviderCard accessibility (role="radio", aria-checked, keyboard handler). Clean hook separation. Pricing data well-structured.
