## Story
**As a** advogado (lawyer),
**I want** cadastrar e acompanhar processos judiciais,
**so that** eu possa gerenciar todos os casos do escritório em um só lugar.

## Acceptance Criteria
1. Migration `00007_create_cases.sql` cria tabela `cases` com RLS
2. Pagina `/cases` com tabela: numero do processo, cliente, tribunal, vara, status, advogado responsavel
3. Formulario de cadastro: numero, tribunal (select), vara, assunto, parte contraria, cliente (select), advogado responsavel (select)
4. Status do processo: ativo, arquivado, encerrado
5. Filtros por status, tribunal, advogado
6. Busca por numero do processo ou nome do cliente

### Concerns

1. **MEDIUM — `client_id` join references `profiles` instead of `clients`:** In `src/services/cases.ts` line 32, the query uses `client:profiles!cases_client_id_fkey(full_name)`. Since `client_id` has no FK constraint (comment in migration says "FK to clients table will be added when clients migration is created"), this join hint will fail at runtime if PostgREST cannot resolve the relationship. The `clients` table now exists (from Story 4.1), so the FK should be added and the join should reference `clients` not `profiles`. Same issue in `src/services/caseDetail.ts` line 20.
2. **MINOR — CaseForm does not include "Cliente" Select field populated from clients:** AC3 specifies "Cliente (Select)" but CaseForm.tsx has no `client_id` field rendered in the UI, despite the schema allowing it. The form only has assigned_to (lawyer) Select but no client selector.
3. **MINOR — Search does not cover client name:** AC6 specifies "Busca por numero do processo ou nome do cliente" but the search only queries `case_number.ilike` (line 54 in cases.ts). Client name search is missing.
4. **MINOR — Unit tests not implemented:** Listed in Testing section as pending.
