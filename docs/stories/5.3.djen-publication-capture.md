## Story
**As a** advogado (lawyer),
**I want** receber automaticamente publicações do DJEN relevantes aos meus processos,
**so that** eu não perca prazos processuais importantes.

## Acceptance Criteria
1. Migration `00008_create_publications.sql` cria tabela `publications` com RLS
2. Edge Function `djen-scraper` que busca publicações por nome do advogado
3. Cron job (Supabase pg_cron ou Edge Function schedulada) executando diariamente
4. Página `/cases/publications` com feed de publicações: data, conteúdo, processo vinculado, status lido/não lido
5. Notificação visual (badge no menu) quando há publicações não lidas
6. Vinculação automática publicação → processo quando número do processo é detectado

  ```sql
  CREATE OR REPLACE FUNCTION invoke_djen_scraper()
  RETURNS void AS $$
  DECLARE
    edge_function_url text;
    service_role_key text;
  BEGIN
    SELECT decrypted_secret INTO edge_function_url
    FROM vault.decrypted_secrets WHERE name = 'edge_function_base_url';

    SELECT decrypted_secret INTO service_role_key
    FROM vault.decrypted_secrets WHERE name = 'service_role_key';

    PERFORM net.http_post(
      url := edge_function_url || '/djen-scraper',
      headers := jsonb_build_object(
        'Authorization', 'Bearer ' || service_role_key,
        'Content-Type', 'application/json'
      ),
      body := '{}'::jsonb
    );
  END;
  $$ LANGUAGE plpgsql SECURITY DEFINER;
  ```


### Estratégia de Scraping do DJEN
- O DJEN (Diário da Justiça Eletrônico Nacional) publica em `https://www.in.gov.br/leiturajornal` ou endpoints da API do Diário Oficial da União
- Alternativa: usar a API do portal JusBrasil ou DataJud (CNJ) para busca de publicações
- O scraper deve ser resiliente a mudanças no layout do site — usar seletores genéricos
- Considerar usar a API do DJEN se disponível (preferível a scraping HTML)
- Rate limiting: máximo 1 request por segundo para não sobrecarregar o servidor do DJEN
- Encoding: conteúdo em PT-BR com acentos — garantir UTF-8 em toda a pipeline
- Deduplicação: usar `external_id` (hash do conteúdo + data + fonte) como unique constraint

### Supabase Realtime para Notificações
- Habilitar Realtime na tabela `publications` via migration SQL
- No frontend, usar `supabase.channel('publications')` com filtro por `organization_id`
- Eventos: `INSERT` (nova publicação capturada), `UPDATE` (marcada como lida)
- Cleanup obrigatório no unmount: `supabase.removeChannel(channel)`
- Padrão de uso:
  ```typescript
  const channel = supabase
    .channel('publications-realtime')
    .on('postgres_changes', {
      event: 'INSERT',
      schema: 'public',
      table: 'publications',
      filter: `organization_id=eq.${orgId}`
    }, (payload) => {
      queryClient.invalidateQueries({ queryKey: ['publications'] });
      queryClient.invalidateQueries({ queryKey: ['unread-publications-count'] });
    })
    .subscribe();
  ```

### pg_cron e Vault
- `pg_cron` executa dentro do PostgreSQL — ideal para agendar chamadas à Edge Function
- Usar `net.http_post` (extensão `pg_net`) para chamar a Edge Function
- Segredos (URL da Edge Function, service_role_key) armazenados no Supabase Vault
- O cron job roda apenas em dias úteis (seg-sex) às 07:00 BRT (10:00 UTC)
- Para debug: executar manualmente via `SELECT invoke_djen_scraper();`

### Formato do Número de Processo CNJ
- Regex: `/\d{7}-\d{2}\.\d{4}\.\d\.\d{2}\.\d{4}/g`
- Exemplo: `0001234-56.2024.8.17.0001`
- Estrutura: NNNNNNN-DD.AAAA.J.TR.OOOO (N=sequencial, D=dígito verificador, A=ano, J=justiça, TR=tribunal, O=origem)
