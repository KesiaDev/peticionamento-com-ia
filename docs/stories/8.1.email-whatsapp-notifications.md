## Dev Notes

### Arquitetura das Edge Functions

Ambas as Edge Functions seguem o padrao Deno/Supabase existente no projeto (`supabase/functions/`). Cada funcao reside em seu proprio diretorio com um `index.ts`.

**Estrutura padrao de uma Edge Function:**

```typescript
// supabase/functions/email-notify/index.ts
import { serve } from "https://deno.land/std@0.168.0/http/server.ts";
import { createClient } from "https://esm.sh/@supabase/supabase-js@2";
import { Resend } from "npm:resend";

const resend = new Resend(Deno.env.get("RESEND_API_KEY"));

serve(async (req) => {
  // 1. Validar Authorization header
  // 2. Parsear e validar body (type, recipient_email, data)
  // 3. Consultar notification_preferences do usuario
  // 4. Se email habilitado e tipo habilitado, selecionar template
  // 5. Enviar via Resend
  // 6. Retornar resultado
});
```

**Estrutura da Edge Function WhatsApp:**

```typescript
// supabase/functions/whatsapp-notify/index.ts
import { serve } from "https://deno.land/std@0.168.0/http/server.ts";
import { createClient } from "https://esm.sh/@supabase/supabase-js@2";

interface WhatsAppProvider {
  sendMessage(phone: string, message: string): Promise<{ success: boolean; messageId?: string }>;
}

// Z-API adapter
class ZApiProvider implements WhatsAppProvider {
  async sendMessage(phone: string, message: string) {
    const response = await fetch(`${Deno.env.get("ZAPI_URL")}/send-text`, {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
        "Client-Token": Deno.env.get("ZAPI_TOKEN")!,
      },
      body: JSON.stringify({ phone, message }),
    });
    const data = await response.json();
    return { success: response.ok, messageId: data.messageId };
  }
}

// Evolution API adapter
class EvolutionApiProvider implements WhatsAppProvider {
  async sendMessage(phone: string, message: string) {
    const instance = Deno.env.get("EVOLUTION_INSTANCE");
    const response = await fetch(
      `${Deno.env.get("EVOLUTION_URL")}/message/sendText/${instance}`,
      {
        method: "POST",
        headers: {
          "Content-Type": "application/json",
          apikey: Deno.env.get("EVOLUTION_API_KEY")!,
        },
        body: JSON.stringify({ number: phone, textMessage: { text: message } }),
      }
    );
    const data = await response.json();
    return { success: response.ok, messageId: data.key?.id };
  }
}

function getProvider(): WhatsAppProvider {
  const provider = Deno.env.get("WHATSAPP_PROVIDER") || "zapi";
  return provider === "evolution" ? new EvolutionApiProvider() : new ZApiProvider();
}

serve(async (req) => {
  // 1. Validar Authorization header
  // 2. Parsear e validar body
  // 3. Consultar notification_preferences
  // 4. Se whatsapp habilitado, formatar mensagem por tipo
  // 5. Enviar via provider selecionado
  // 6. Retornar resultado
});
```

### Resend SDK para Deno

- Importar como `npm:resend` nas Edge Functions do Supabase (Deno runtime suporta npm specifiers)
- Env var `RESEND_API_KEY` configurada nos secrets da Edge Function
- Dominio de envio deve ser verificado no painel do Resend (ex: `noreply@juristech.com.br`)
- Resend suporta templates HTML inline -- nao precisa de servico externo de templates

### WhatsApp: Z-API vs Evolution API

| Aspecto | Z-API | Evolution API |
|---------|-------|---------------|
| Tipo | SaaS (pago) | Self-hosted (open source) |
| Setup | Criar conta + conectar WhatsApp | Deploy proprio + conectar WhatsApp |
| Custo | Planos mensais | Gratuito (custo de infra) |
| Documentacao | https://developer.z-api.io | https://doc.evolution-api.com |
| Ideal para | MVP/inicio rapido | Producao com controle total |

A decisao de qual usar fica com o operador do sistema. O adapter pattern permite trocar sem alterar codigo.

### Preferencias de Notificacao

A tabela `notification_preferences` e separada de `profiles` para manter responsabilidade unica. Relacao 1:1 com `profiles` via `profile_id UNIQUE`.

**Opcoes de preferencia por usuario:**
- Canal: email, WhatsApp, ambos, nenhum
- Tipos: publicacoes novas, prazos proximos, tarefas atribuidas
- Antecedencia de prazo: configuravel em horas (default 48h)
- Telefone WhatsApp: armazenado na tabela de preferencias (nao em profiles)

### Code Quality

- **Strengths:** Adapter pattern for WhatsApp providers is well-designed and extensible. Preference-checking in both Edge Functions prevents unwanted notifications. Service layer handles "no row found" gracefully by auto-creating defaults. Phone mask formatting is simple and effective.
- Email templates are well-styled with proper inline CSS (email-safe).
- `hasChanges` dirty check prevents unnecessary saves.

### Security

- Both Edge Functions validate Authorization header and verify user via `getUser()`.
- Service role client used only for preference lookup (not for user-facing operations). API keys (RESEND_API_KEY, WHATSAPP_API_TOKEN) stored as env vars, never exposed to frontend.
- **CONCERN:** `email-notify` uses ANON_KEY for user auth but SERVICE_ROLE_KEY for preferences lookup. This is correct architecture but creates two Supabase clients per request -- acceptable but worth noting.
- **CONCERN:** WhatsApp `to` field comes from request body, not from stored preferences. Caller could send to arbitrary numbers. Consider validating `to` matches `prefs.whatsapp_number` in production.
- CORS `*` on both Edge Functions -- acceptable for server-to-server but should be tightened for production.

### Edge Functions

- Error handling with try/catch and structured JSON error responses.
